# Pain Memory - Predator（捕食者）実験 設計記録

## これは何の実験か

やかん（一瞬の火傷）・毒（薬物動態モデルによる持続ダメージ）に続く第3弾。今までの2つは、危険が固定された座標にあり「その場で観察すれば安全かどうか分かる」という共通点があった。捕食者は**位置が動く危険**であり、質的に違う。「今見て確認する」だけでなく「次にどう動くか予測する」という、別種の認識が必要になる可能性がある、というのが今回の仮説。

## 4つのAIへの相談

Grok、Google Gemini、ChatGPT、ディープリサーチの4つに、同一のプロンプトで設計相談を行った。

### 各AIの提案要旨

**ChatGPT**：段階を分けて難易度を上げる進め方（①固定パトロール→②ランダム歩行→③追跡型）を提案。状態には捕食者の直前の移動量を追加。**「Oracle（未来を完全に見通せる）」条件を比較に加え、Pain Memoryの限界と認識能力の限界を切り分ける**という提案が独自。意地悪設計は「見た目では判断できない囮（80%危険・20%安全）」を最重要とした。グリッドは7×7で十分という立場。

**Google Gemini**：固定パトロール（3×3を時計回り）を推奨、状態は全域可視を基本形とする。**「Pain Memoryの減衰速度と、捕食者の巡回周期が同期してしまう」という新しい比較軸**を提案。意地悪設計として「逆位相パトロール（減衰周期を悪用した罠）」を提示。グリッドは7×7で十分という立場。

**Grok**：確率的追跡＋慣性のハイブリッド（70%の確率でエージェントへ向かう＋60%の確率で直前の方向を維持）を推奨。**近距離（2.5以内）は正確に見え、遠距離は方向だけ分かる、という視野の限定**を導入。意地悪設計は4種類（隠れて急接近、複数の捕食者、行動に応じて反応が変わる捕食者、持続追跡ダメージ）。グリッドは7×7で十分という立場。

**ディープリサーチ**：4者の中で唯一、**「7×7は不適切で、15×15か20×20に広げるべきだ」と明確に主張**。理由は、7×7では最大距離が12歩しかなく、迂回ルートを学習する前に追い詰められて終わる、というもの。エゴセントリック（自己中心の相対座標）で状態数を抑える案を提示。捕食者はパトロール＋追跡の2状態モデル、視野制限（POMDP）を採用。**新しい評価指標を4つ**（ゴール到達効率、行動の多様性、予測的回避率、記憶解除後の再探索の速さ）を提案。意地悪設計の一つ「共鳴トラップ」は、Geminiが独立に提案した「減衰周期の同期」と本質的に同じ着想。

## 書記官（Claude）の統合案

**強く採用する点（複数AIが独立に一致）**：
1. 段階を分けて難易度を上げる進め方（固定パトロール→予測が要る状況）
2. **減衰の周期と、危険の再来の周期を意図的に一致させる「共鳴トラップ」**を最優先の意地悪設計とする（Gemini・ディープリサーチが独立に到達した着想であるため）
3. 視野の限定（近くは正確に見える、遠くは方向のみ／不明）を導入し、Grokの連続的な距離区切り方式を採用する

**判断が割れた点への対処（グリッドサイズ）**：3対1でディープリサーチのみが拡張を主張。多数決では決めず、まず7×7で組んで「迂回する前に追い詰められて終わる」という兆候が実際に出るか診断し、出れば拡張する。理屈ではなく実測で決着をつける（これまでのP_local方式決着実験と同じ方法論）。

## 実装セッション1（2026年7月8日）の記録

### 発見・修正したバグ
タブー判定・痛みの記憶のキーが、**行動（どちらの方向に動くか）を考慮せず、状態だけで一律に判定していた。** やかん・毒では「危険な行動が1種類だけ（GRAB/ACT）」だったため状態単位で十分だったが、捕食者は「どの方向に動くかで危険かどうかが変わる」ため、移動先（行動の結果）をキーに含める必要があった。修正済み（`predator_v01_full.py`）。

### 予備結果（10シード、`predator_v01_results_10seed.json`）
修正後、**やかん・毒とは正反対のパターン**が出た。PermBan（認識あり・なし問わず）が、他の2方式より衝突回数が少なく、成績も良い。

| 条件 | 切替前平均 | 切替直後50Ep | 末尾100Ep平均 | 前半衝突回数 |
|---|---|---|---|---|
| PermBan | 8.50 | 5.85 | 9.22 | 10.3 |
| NoMemory | 8.02 | 6.42 | 8.28 | 30.9 |
| FCA | 8.29 | 6.46 | 8.78 | 19.3 |
| PermBan+認識 | 3.38 | -4.76 | 8.34 | 10.5 |
| NoMemory+認識 | 4.25 | -9.38 | 8.34 | 80.7 |
| FCA+認識 | 4.68 | -8.03 | 8.30 | 69.5 |

### 未解決の疑問（次回の最優先課題）
これは実装バグではなく、**構造上の違いによる可能性がある。** やかん・毒は「目的達成の手段が1つだけ」で、それを永久禁止すると詰んだ。今回の捕食者実験は「ゴールに着けば良い、経路はいくらでもある」構造のため、パトロール経路周辺のマスを永久禁止しても単に迂回できてしまい、PermBanが崩壊するボトルネックが存在しない可能性がある。

次回、この仮説を検証する（例えば、ゴールへの経路を1本の隘路に絞る地形にする等）。まだ確認していない。

## v02（2026年7月8日）— 3つの失敗を修正し、やかんの結果を動く危険で再現

### v01の3つの失敗（前セッションで特定）への対処
1. 危険が機能しない → 壁で「短い危険路(6歩) vs 長い安全路(12歩)」を作り、危険を避けられなくした
2. 状態爆発（960状態）→ 捕食者情報を「行＋移動方向」に圧縮（257状態に収まった）
3. ボトルネックなし → 2ゲート構造で制約

### 設計を確定させるまでの試行錯誤（3版）
- 第1版：捕食者を通り道(x=2)、Phase Bでy=5,6へ → PermBanが両ゲートの通り道を全禁止して完全に詰んだ
- 第2版：捕食者を出口側(x=4)へ → gap_midの遠い出口だけ禁止され、PermBanが袋小路にはまった
- 最終版：捕食者を入口側(x=2)、gap_top(y=6経由)を捕食者が絶対に触れない「常時安全な逃げ場」にした。これでやかんの「ミトン＝常に安全な道」と同じ構造になった

### 発見・修正したバグ
衝突時にエージェントが跳ね返る設計にしたため、記録すべき「危険な移動先」ではなく「跳ね返った後の元位置」を記録していた。タブー判定は「移動しようとした先」で見ているのに記録が噛み合わず、3方式が全て同一結果になっていた（v01と同じ症状）。移動しようとした先(intended)で記録するよう修正。

### 確認できたこと（重要）
**「認識なし」条件が、やかん実験と全く同じ綺麗な結果になった。**
- PermBan：安全（衝突5.9）だが硬直——最短路midを再発見できず、遠回りtopのまま（報酬9.23）
- NoMemory：適応するが危険（最短路midだが衝突57.4）
- FCA：適応＋安全（最短路midを使いつつ衝突30.9で半分）＝**両立**

FCAの優位性は「最終報酬」ではなく「衝突の少なさ（学習中の安全性）」に表れる。これはやかんの「火傷が少ない」と同じ構造。危険の性質が「固定」から「移動」に変わっても、この関係が崩れなかった。

### 未解決の点
「認識あり」条件でPermBanが完全に詰んだ（成功率1%）。これは物理的封鎖ではなくQ学習の探索問題——捕食者の位置を状態に加えたことで状態空間が広がり、禁止された行動が探索から除外されるPermBanだけが、代替路(gap_top)への報酬信号を得られず学習しきれなかった。gap_top迂回路は禁止ゼロで通行可能なのに学習できていない。やかんでPermBanが緩やかな劣化(2.75)で済んだのは、安全な道が「別目的地」として報酬付きで学習できたため。この掘り下げは次回に持ち越し。
